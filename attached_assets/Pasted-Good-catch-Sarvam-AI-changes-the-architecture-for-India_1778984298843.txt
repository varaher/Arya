## The Updated Language Detection Logic

```typescript
// language-resolver.ts

interface LanguageSource {
  source: 'sarvam_stt' | 'text_detection' | 'user_preference' | 'profile';
  language_code: string;
  confidence: number;
}

async function resolveLanguage(
  input: {
    type: 'voice' | 'text';
    text?: string;
    sarvam_detected?: string;    // from Sarvam STT response
    sarvam_confidence?: number;
    user_id: string;
  }
): Promise<string> {

  if (input.type === 'voice' && input.sarvam_detected) {
    // Sarvam already detected it — trust it
    // Sarvam's language detection is more accurate than
    // script-based text detection for Indian languages
    await updateLanguageProfile(
      input.user_id,
      input.sarvam_detected,
      'sarvam_stt'
    );
    return input.sarvam_detected;
  }

  if (input.type === 'text' && input.text) {
    // Text input — use script detection
    const detected = detectLanguage(input.text);
    await updateLanguageProfile(
      input.user_id,
      detected,
      'text_detection'
    );
    return detected;
  }

  // Fallback to stored preference
  const profile = await getLanguageProfile(input.user_id);
  return profile.preferred_response_language || 'en';
}
```

## The Cost Architecture — Sarvam vs Global Languages

This is important for your budget.

```
Indian language voice:
  Sarvam STT      → transcription
  Claude          → thinking + response
  Sarvam TTS      → speaking back
  Cost:           Sarvam pricing + Claude API

Global language voice (Arabic, French, Spanish etc):
  Web Speech API  → transcription (FREE, browser-based)
  Claude          → thinking + response (detects language)
  No TTS          → text response only
  OR
  Browser TTS     → basic voice (free, lower quality)
  Cost:           Claude API only

Indian languages get the premium Sarvam experience.
Global languages get functional but simpler voice.
This is the right trade-off — your core market
gets the best experience.
```

## One Thing To Check In Your Codebase

You mentioned Sarvam is already integrated for STT. Check whether the Sarvam API response includes the detected language code — it does in most configurations.

```typescript
// Current Sarvam STT call probably looks like:

  method: 'POST',
  body: formData
});

const result = await sarvamResponse.json();
// result.transcript → the transcribed text
// result.language_code → CHECK IF THIS EXISTS
```

If `language_code` is in the Sarvam response, your developer saves writing the script detection for voice input entirely — Sarvam handles it.

## Summary — What Sarvam Changes

```
Detection:     Sarvam STT replaces script detection for voice input
               Script detection still used for typed text

Output:        Sarvam TTS speaks Indian language responses
               Global languages get text-only (for now)

Quality loop:  Same thumbs up/down system
               But applied to transcript card below audio

Cost:          Sarvam cost per voice minute
               Claude cost per response
               Cache hits reduce Claude cost
               Voice responses always need Sarvam TTS
               (no way to cache audio — text cached, re-spoken)
```

The flow you designed is correct. Sarvam sits at the edges — input transcription and output speech. Claude stays in the middle doing the thinking. The quality loop, cache, and learning engine work exactly the same way regardless of whether input came from voice or text.
